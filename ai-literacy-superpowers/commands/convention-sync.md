# /convention-sync

Sync the project's HARNESS.md conventions and constraints to
tool-specific convention files for Cursor, Copilot, and Windsurf.

Read the `convention-sync` skill from this plugin before proceeding.
It provides the output format reference and conflict resolution rules.

## Process

### 1. Read HARNESS.md

Read `HARNESS.md` from the project root. If it does not exist, stop
and tell the user:

> No HARNESS.md found in the project root. Run `/harness-init` to
> create one, then re-run `/convention-sync`.

### 2. Parse Context and Constraints

Extract the following from HARNESS.md:

- **Context > Stack** — languages, build system, test framework,
  container strategy
- **Context > Conventions** — naming, file structure, error handling,
  documentation
- **Constraints** — every constraint heading with its Rule, Enforcement,
  Tool, and Scope fields

### 3. Generate Convention Files

For each supported tool, generate the convention files described in the
`convention-sync` skill's Output Format Reference:

#### Cursor

- Create `.cursor/rules/` directory if it does not exist
- Generate `.cursor/rules/conventions.mdc` with frontmatter
  (`description`, `globs: **/*`, `alwaysApply: true`) and the Context
  section content
- Generate `.cursor/rules/constraints.mdc` with frontmatter and one
  section per constraint

#### Copilot

- Create `.github/` directory if it does not exist
- Generate `.github/copilot-instructions.md` combining Context and
  Constraints in flat markdown with no frontmatter

#### Windsurf

- Create `.windsurf/rules/` directory if it does not exist
- Generate `.windsurf/rules/conventions.md` with Context section content
  (no frontmatter)
- Generate `.windsurf/rules/constraints.md` with one section per
  constraint (no frontmatter)

Each generated file must include a header comment or note:

> These conventions are synced from HARNESS.md. Do not edit this file
> directly — run `/convention-sync` to regenerate.

### 4. Handle Existing Files

Before writing each file, check whether it already exists:

- If the file does not exist, create it and record status as **created**
- If the file exists and matches the generated content exactly, record
  status as **unchanged**
- If the file exists but differs from the generated content, show the
  user what changed and ask whether to overwrite:
  - If the user approves, write the file and record status as **updated**
  - If the user declines, skip the file and record status as **skipped**

### 5. Verify

After generating all files, verify that every constraint heading from
HARNESS.md appears in each output file. Report any missing constraints
as errors.

### 6. Report

Show a summary table:

```text
| Tool     | File                                 | Status    |
| -------- | ------------------------------------ | --------- |
| Cursor   | .cursor/rules/conventions.mdc        | created   |
| Cursor   | .cursor/rules/constraints.mdc        | created   |
| Copilot  | .github/copilot-instructions.md      | updated   |
| Windsurf | .windsurf/rules/conventions.md       | created   |
| Windsurf | .windsurf/rules/constraints.md       | unchanged |
```

Tell the user how many files were created, updated, unchanged, or
skipped. If any constraints were missing from output files, list them.
