## Constraint Assessment

### Release traceability

> Every version in `plugin.json` must have a matching
> `## X.Y.Z — YYYY-MM-DD` heading in `CHANGELOG.md` and a `vX.Y.Z`
> git tag.

#### Falsifiability scoring

| Question | Answer |
| --- | --- |
| What do you verify? | (1) Changelog heading matches plugin.json version at PR time. (2) vX.Y.Z git tag exists for every published version. |
| What counts as evidence? | CI check output from version-check.yml, git tag list, auto-tag.yml run logs, GC rule run logs. |
| What happens on failure? | PR is blocked (changelog mismatch). Tag is auto-created (GC finding or auto-tag workflow). |

**Score: Falsifiable.** All three falsifiability questions have concrete,
verifiable answers. The constraint specifies deterministic enforcement
with defined tools, scopes, and failure actions.

## Prioritised Recommendations

1. **Continue exercising the governance audit cadence.** This re-audit
   confirms the constraint remains healthy after significant version
   growth (8 versions in a single day). The next governance audit
   should be on or before 2026-07-15 unless new governance constraints
   are added sooner.

2. **Consider governance constraints for other harness areas.** The
   project now has 13 constraints and 13 GC rules but only 1
   governance constraint. Candidates for governance promotion:
   - **No secrets in source** — has compliance implications
   - **Tests must pass** — currently "unverified" enforcement;
     promoting to deterministic with governance fields would
     strengthen the quality gate
   - **Spec-first commit ordering** — captures process governance
     around change management

3. **Monitor the terminology gap.** The constraint uses both
   "published plugin versions" (governance requirement) and "version
   in plugin.json" (rule). While operationally equivalent today,
   if the project ever separates "published" from "in plugin.json"
   (e.g. draft versions, pre-release versions), this could become
   a Stage 1 terminology drift. No action needed now — flag for
   future audits.

4. **Formalise the GC rule execution schedule.** The release-tag-
   completeness GC rule is declared but its execution depends on
   manual /harness-gc invocation. Consider adding it to an automated
   cadence (e.g. weekly CI job) to close the loop fully.
