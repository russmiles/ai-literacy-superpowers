# HARNESS.md Format

Section-by-section reference for the `HARNESS.md` file. This file is
the source of truth for your project's harness — conventions,
constraints, garbage collection rules, and observability settings.

Generated by `/harness-init` from the template at
`templates/HARNESS.md`. Edit freely after generation.

---

## File Structure

```text
# Harness — {{PROJECT_NAME}}
<!-- template-version: X.Y.Z -->

## Context
  ### Stack
  ### Conventions

## Constraints
  ### [constraint name] (one per constraint)

## Garbage Collection
  ### [rule name] (one per rule)

## Observability
  ### Operating cadence
  ### Health thresholds
  ### Regression detection

## Status
```

The `<!-- template-version: X.Y.Z -->` comment tracks which plugin
version generated the template. The `/harness-upgrade` command
compares this against the installed plugin version to detect new
template content.

---

## Context

Primes LLMs and new team members with the information needed to work
effectively in the codebase. Content here feeds into agent context
windows and convention sync files.

### Stack

Four required fields:

| Field | Description |
| --- | --- |
| **Primary languages** | Languages used in the project, with context for each |
| **Build system** | Build tool or "None" for non-compiled projects |
| **Test framework** | Test runner or "None" |
| **Container strategy** | Docker/containerisation approach or "None" |

### Conventions

Four required fields. Each must be concrete enough that a reviewer
could objectively verify compliance:

| Field | Description |
| --- | --- |
| **Naming** | Casing, patterns, file naming conventions |
| **File structure** | How files are organised — by type, by feature, etc. |
| **Error handling** | How errors are handled and propagated |
| **Documentation** | What must have docs, what comments should explain |

## Constraints

Each constraint is a verification slot backed by a deterministic
tool, an agent-based review, or declared but not yet automated.

### Required fields

| Field | Description | Values |
| --- | --- | --- |
| **Rule** | Precise, enforceable statement of what must be true | Free text |
| **Enforcement** | How the constraint is verified | `deterministic`, `agent`, `unverified` |
| **Tool** | The command or agent that checks it | Command string, agent name, or `none yet` |
| **Scope** | When the check runs | `commit`, `pr`, `weekly`, `manual` |

### Governance constraint fields (optional)

When a constraint has compliance or policy implications, add these
fields after the four required fields:

| Field | Description |
| --- | --- |
| **Governance requirement** | The regulation, policy, or standard being encoded |
| **Operational meaning** | What the requirement means in engineering terms |
| **Verification method** | How compliance is verified (deterministic, agent, manual) |
| **Evidence** | What artefacts demonstrate compliance |
| **Failure action** | What happens when verification fails |
| **Frame check** | Whether engineering, compliance, and AI system interpretations are aligned |

Use `/governance-constrain` for guided authoring.

### Enforcement enum

| Value | Meaning |
| --- | --- |
| `deterministic` | A tool runs automatically and produces a pass/fail result |
| `agent` | An LLM agent reviews the change and reports findings |
| `unverified` | Declared intent only — no automated verification yet |

### Scope enum

| Value | When it runs | Use for |
| --- | --- | --- |
| `commit` | Advisory warning during editing (PreToolUse hooks) | Fast checks: linting, syntax, secrets |
| `pr` | Strict CI gate at merge time | Comprehensive checks: tests, spec compliance |
| `weekly` | Scheduled periodic sweep (GC workflow) | Slow or expensive checks |
| `manual` | On-demand via `/harness-audit` | Checks that need human judgement on timing |

## Garbage Collection

Periodic checks that fight entropy — the slow drift that neither
real-time hooks nor PR gates catch.

### Required fields

| Field | Description | Values |
| --- | --- | --- |
| **What it checks** | Concrete description of the entropy being detected | Free text |
| **Frequency** | How often the check runs | `weekly`, `monthly`, `quarterly` |
| **Enforcement** | Whether the check is deterministic or needs LLM judgement | `deterministic`, `agent` |
| **Tool** | The command or agent that runs the check | Command string or `harness-gc agent` |
| **Auto-fix** | Whether the check can fix findings automatically | `true` (with description), `false` |

### Built-in rules

The template includes starter rules. Projects can add, remove, or
modify rules freely. See the
[Garbage Collection]({% link plugins/ai-literacy-superpowers/garbage-collection.md %})
explanation page for the design rationale.

### Optional rule categories

The template includes commented-out sections for:

- **Governance GC** — governance constraint freshness, semantic drift
  early warning, governance debt cycle check
- **Fitness functions** — layer boundary compliance, complexity
  hotspots, coupling trend, dependency age budget

Uncomment the categories relevant to your project.

## Observability

Controls how often observability activities run and what thresholds
trigger health status changes. See
[Configure Observability]({% link plugins/ai-literacy-superpowers/configure-observability.md %})
for a guide to tuning these settings.

### Snapshot cadence

```text
- Snapshot cadence: monthly
```

Options: `weekly` (staleness threshold: 10 days), `fortnightly`
(21 days), `monthly` (30 days). Weekly is recommended for
Observatory corpus projects.

### Operating cadence

Target frequencies for each observability activity.
Meta-observability checks compare actual run dates against these
targets.

| Activity | Default | Threshold |
| --- | --- | --- |
| Harness audit (`/harness-audit`) | quarterly | 90 days |
| AI literacy assessment (`/assess`) | quarterly | 90 days |
| Reflection review and promotion | monthly | 30 days |
| Cost capture (`/cost-capture`) | quarterly | 90 days |

### Health thresholds

Thresholds for the aggregate health status in each snapshot's Meta
section.

| Threshold | Default | Meaning |
| --- | --- | --- |
| Minimum enforcement ratio for Healthy | 70% | Below this, health drops to Attention |
| Consecutive zero-finding GC snapshots before alert | 3 | GC may be misconfigured if nothing found for 3 snapshots |
| Unpromoted reflection age before learning flow is stalled | 60 days | Reflections exist but none promoted — learning loop broken |
| Consecutive declining trend snapshots before alert | 3 | A metric declining 3 times in a row signals a real problem |

### Regression detection

Conditions that trigger the regression flag in each snapshot's
Regression Indicators section.

| Threshold | Default | Meaning |
| --- | --- | --- |
| Cadence non-compliance | 2 or more activities overdue | One overdue is normal; two signals systemic slowdown |
| Reflection drought | 4 consecutive weeks with zero reflections | The compound learning loop has stopped |

See [Regression Detection]({% link plugins/ai-literacy-superpowers/regression-detection.md %})
for the conceptual explanation.

## Status

Auto-updated by `/harness-audit`. Do not edit manually.

```text
Last audit: YYYY-MM-DD
Constraints enforced: N/M
Garbage collection active: N/M
Drift detected: yes/no
```

| Field | How it's computed |
| --- | --- |
| Last audit | Date of the most recent `/harness-audit` run |
| Constraints enforced | Count of constraints with enforcement = `deterministic` or `agent`, over total |
| Garbage collection active | Count of GC rules with enforcement != `none`, over total |
| Drift detected | Whether the auditor found mismatches between declarations and reality |
